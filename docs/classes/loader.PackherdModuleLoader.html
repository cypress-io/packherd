Class PackherdModuleLoader

The PackherdModuleLoader is responsible for resolving required/imported modules. To do that in an efficient manner it first looks inside the Node.js Module cache and then tries to load a module either from the provided moduleExports then the moduleDefinitions and only then from the file system via the Node.js module loading mechanism.

tryLoad

  • tryLoad(moduleUri: string, parent: undefined | NodeModule, isMain: boolean): ModuleLoadResult
    • Defined in src/loader.ts:521

    Here we try to load the module for the given URI as efficiently as possible. Thus we attempt the below steps in the given order:

    1. Load directly from the Node.js module cache providing the moduleUri (this is what Node.js would do as well)
    2. Attempt to resolve the module key
    3. If we found a module key that is a full path try to load from the Node.js cache again, this time using the that full path
    4. Try to resolve the module's full path via path.resolve
    5. Try the Node.js cache again with that full path

    At this point we give up trying to find this module in the Node.js cache some how ..

    1. Try to load the module either from moduleExports or instantiate it from one of the moduleDefinitions
    2. Resolve the module via the Node.js resolution mechanism (requires I/O)
    3. Derive another moduleKey from the resolved path and try moduleExports or moduleDefinitions again in order to at least avoid having to read the module content from the file system and ideally also the module initialization overhead
    4. We failed miserable and have to call the original Node.js module loader which requires I/O and has the maximum overhead

    NOTE: that we track the module load in each case.

    Parameters

    • moduleUri: string

      the uri as specified in the require or import statement used for diagnostics

    • parent: undefined | NodeModule

      the module.parent

    • isMain: boolean

      if true this is the app entrypoint, i.e was launched via node|electron entrypoint

    Returns ModuleLoadResult

    info regarding how the module was resolved, the origin from which it was loaded, it's exports and the fullPath` resolved for it

tryResolve

  • tryResolve(moduleUri: string, opts?: GetModuleKeyOpts): ModuleResolveResult
    • Defined in src/loader.ts:447

    Given the moduleUri it attempts to resolve the full path to the module. The idea here is to only fall back to Node.js resolution which requires access to the file system when all other attempts fail.

    To that end we try the following in the given order:

    1. resolve via this.getModuleKey given it returns a moduleKey that is a full path
    2. expand the path to a full path using path.resolve (still no I/O required)
    3. fall back to Node.js resolution mechanism (this requires I/O)

    Parameters

    • moduleUri: string

      the uri as specified in the require or import statement used for diagnostics

    • Optional opts: GetModuleKeyOpts

      control how the module key is obtained

    Returns ModuleResolveResult
